  QTBot — 项目概览

  这是一个基于 NoneBot2 框架的 QQ 聊天机器人，专门为游戏 Project QT（PQT）提供
  Wiki 数据查询服务，使用 Python 编写。

  核心架构

  ┌────────────┬───────────────────────────────────────────────────────────┐
  │     层     │                         技术/组件                         │
  ├────────────┼───────────────────────────────────────────────────────────┤
  │ 机器人框架 │ NoneBot2 + nonebot-adapter-qq                             │
  ├────────────┼───────────────────────────────────────────────────────────┤
  │ 数据存储   │ SQLite（本地）+ JSON 文件 / Google Sheets（可选外部存储） │
  ├────────────┼───────────────────────────────────────────────────────────┤
  │ 依赖注入   │ dependency-injector 容器管理                              │
  ├────────────┼───────────────────────────────────────────────────────────┤
  │ 数据来源   │ Project QT Wiki 爬取                                      │
  └────────────┴───────────────────────────────────────────────────────────┘

  主要功能

  1. 角色数据查询（普通用户）

  通过 @机器人 /查询 [角色名] 或 @机器人 /wiki [角色名] 触发，支持：
  - 基础查询 — 返回角色的满星技能、觉醒被动、天赋数据
  - 别名映射 — 支持中英文名和配置的别名查询
  - 参数化查询 —
  可指定星级、技能等级、天赋等级、皮肤序号/等级、背景故事，如：/查询 
  雅典娜-3星-皮肤1:2
  - 翻译 — 追加 :tl 参数调用百度翻译将英文数据译为中文

  2. Wiki 数据爬取（超级用户）

  通过 @机器人 /su ... 触发，支持：
  - 批量爬取（按类型/数量）：/su 抓取-女神-10条数据
  - 单角色爬取：/su 抓取-[角色名]
  - 增量更新：/su 更新:name:skins,arena_skill,...
  - 抓取最新数据（数据库中未收录的词条）

  3. 数据同步（超级用户）

  - 本地 JSON ↔ 数据库 双向同步
  - 可选 Google Sheets 作为外部存储实现多人协作
  - 使用 MD5 哈希判断数据是否有变更

  数据模型

  角色实体包含：名称、别名、头像、竞技场技能（1-8星描述）、觉醒被动、天赋树、皮
  肤、背景故事、所属社团/元素/年级、羁绊、爱好、标签等。

  部署方式

  - 需在 QQ 开放平台申请机器人，配置 .env.prod 中的 bot 凭证
  - 支持 Docker 部署（提供了 Dockerfile + docker-compose.yml）
  - 有 GitHub Actions 自动构建镜像

  目标是：系统启动时，把 characters.json 里的数据同步到 SQLite 数据库。

  具体需要两样东西：

  1. 同步逻辑（AppContainer.init_data）

  在容器里加一个 init_data() 方法，它做两件事：
  - 备份：在同步前，把 data/characters.json 复制一份命名为
  characters_20260706_xxxxxx.json，防止同步出问题丢数据
  - 同步：调用 storage_service.sync_data()，这个方法会读取 characters.json，用
  MD5 哈希对比数据库里的数据，把有变更的更新进去

  2. 触发时机（bot.py 的 on_startup）

  需要一个地方在机器人启动时自动调用上面的 init_data()。NoneBot2 提供了
  driver.on_startup
  生命周期钩子——在框架启动完成后、开始接收消息前执行，这个时机最合适。

  不加的话，新增的 characters.json 数据不会自动进数据库，用户得手动发 /su 
  同步至数据库 指令才行。

     模糊搜索功能

     Context

     当前 /查询 指令使用 NameMapper.get_canonical_name()
     做精确匹配，输入名称不完整或略有偏差时直接返回"未查找到数据"，体验不好。需
     要在精确匹配失败时，根据输入做模糊搜索，返回最多 5 个可能的结果供用户选择。

     改动方案

     1. utils/name_mapper.py — 新增 fuzzy_search 方法

     在 NameMapper 类中添加方法：

     fuzzy_search(query: str, max_results: int = 5) -> List[str]

     逻辑：
     - 遍历 self.data，对每个角色的 name（标准名）和
     aliases（别名列表）做大小写不敏感的 substring 匹配
     - 同一个角色可能有多个别名匹配，只保留一个，按角色去重
     - 按匹配到的顺序取前 max_results 个，返回标准名列表

     2. plugins/wiki.py — 精确匹配失败时调用模糊搜索

     在 handle_wiki 函数中，当前逻辑：

     canonical_name = name_mapper.get_canonical_name(instructions.name)
     if not canonical_name:
         await wiki_cmd.finish(MessageSegment.text(f"未查找到[{instructions.name
     }]的数据，请通知管理员录入"))

     改为：

     canonical_name = name_mapper.get_canonical_name(instructions.name)
     if not canonical_name:
         suggestions = name_mapper.fuzzy_search(instructions.name)
         if suggestions:
             msg = f"未找到「{instructions.name}」，您是不是想找：\n" +
     "\n".join(f"- {s}" for s in suggestions)
         else:
             msg = f"未查找到[{instructions.name}]的数据，请通知管理员录入"
         await wiki_cmd.finish(MessageSegment.text(msg))

     涉及文件

     - utils/name_mapper.py — 新增 fuzzy_search 方法
     - plugins/wiki.py — 修改 handle_wiki 中精确匹配失败的分支

     验证

     1. 找一个不完整名称查询，如 /查询 雅，应返回包含"雅典娜"等建议列表
     2. 找一个完全不存在的名称查询，如 /查询 xyzabc，应返回原有无数据提示
     3. 确保精确匹配仍正常工作

⏺ 这是个常见问题。requests 是纯 HTTP 库，无法处理 Cloudflare 的 JavaScript
  挑战和 TLS 指纹检测。目前有几种成熟方案：

  方案对比

  ┌─────────┬────────────────────────┬─────────────────┬───────────────────┐
  │  方案   │          原理          │    改造成本     │       备注        │
  ├─────────┼────────────────────────┼─────────────────┼───────────────────┤
  │         │ 内置 JS 引擎执行       │                 │ 最常用，但依赖 JS │
  │ cloudsc │ Cloudflare             │ 低，可视为      │  解释器，首次请求 │
  │ raper   │ 的挑战，然后自动携带   │ requests 的平替 │ 略慢              │
  │         │ cookies 重试           │                 │                   │
  ├─────────┼────────────────────────┼─────────────────┼───────────────────┤
  │ curl_cf │ 模拟浏览器 TLS         │ 中，API 与      │                   │
  │ fi      │ 指纹（指纹越像浏览器越 │ requests 类似但 │ 轻量、无需执行 JS │
  │         │ 不容易被拦截）         │ 不完全相同      │                   │
  ├─────────┼────────────────────────┼─────────────────┼───────────────────┤
  │ playwri │                        │ 高，资源消耗大  │ 能应对所有类型验  │
  │ ght     │ 启动真实浏览器内核抓取 │ ，部署要装      │ 证，但太重了      │
  │         │                        │ Chromium        │                   │
  └─────────┴────────────────────────┴─────────────────┴───────────────────┘

  推荐：cloudscraper

  理由是代码里已经大量使用 requests（qt_wiki_crawler.py +
  translation_service.py），而 cloudscraper 的 API 几乎和 requests 完全一致——把
  requests.get(...) 改成 cloudscraper.create_scraper().get(...) 就行，改动最小。
